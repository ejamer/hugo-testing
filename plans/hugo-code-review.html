Hugo Code Review — Fencing-Escrime NB

fenb-1/ site · Hugo v0.161.1 Extended · Theme: Ananke
Reviewed: 2026-05-20 Reviewer: Claude Code (lead developer perspective) Branch: dev
Overall assessment: This is a well-constructed Hugo site that correctly applies many of the framework's key features — data-driven layouts, bilingual routing, Hugo Pipes CSS processing, and no-JS fallbacks. The architecture is coherent and the separation of concerns between templates, data, and content is sound. The main areas for improvement are: leaning on Ananke's CSS pipeline more transparently, eliminating data-entry duplication (manual display dates), hardening the bilingual workflow at the archetype level, and adopting a small number of missing Hugo conventions (deferred scripts, RSS output, `markup` config). None of the identified issues are architectural failures; they are refinements that would improve maintainability and correctness.

1. Configuration (hugo.toml)

Issue canonifyURLs = true — deprecated and conflict-prone

canonifyURLs (line 3) is a post-processing rewrite that absolutizes all relative URLs in the generated HTML. Hugo's own docs call it a "last resort" and it is effectively deprecated. More critically, it conflicts with absURL and relURL template functions, which the site uses throughout. CLAUDE.md even documents a bug this caused: "/pagefind/x.js" | absURL was producing the wrong URL because of the interaction between the base path and this setting. The templates already use relLangURL correctly everywhere — removing canonifyURLs is safe once you verify all links use proper Hugo URL functions.

Fix: Remove canonifyURLs = true from hugo.toml. Run make build, then search the output for any broken relative links. There should be none given the consistent relLangURL / absURL usage.

Warning Production baseURL is commented out

The production URL https://fencingnb.ca/ is commented out (line 1) and the GitHub Pages test URL is active (line 2). This is understandable for a draft site, but it is fragile: a developer could accidentally build for production using the wrong base URL.

Fix: Move the base URL to environment-specific config files. Create config/production/hugo.toml with baseURL = "https://fencingnb.ca/" and config/development/hugo.toml with the test URL. Hugo automatically merges these when --environment production is passed (which make build-prod already does). The single hugo.toml can then omit baseURL entirely or set it to localhost.

Good Language configuration is complete and well-structured

Both [languages.en] and [languages.fr] are fully specified with locale, label, title, and description. Menus are defined per-language with localized names and identical paths, delegating the language prefix to relLangURL in templates — the correct Hugo pattern. The comment on line 91 ("French menu URLs omit the /fr/ prefix; relLangURL in the nav template adds it") is exactly the right approach and shows clear understanding of how Hugo handles multilingual routing.

Warning No [markup] configuration block

Hugo defaults to safe Goldmark rendering, which strips raw HTML from Markdown. The policies section serves PDFs and documents — if any body content ever needs embedded iframes, HTML tables, or raw links, they will be silently stripped. Additionally, table-of-contents settings and code highlight themes are at their defaults.

Fix: Add an explicit [markup] section to make the intent clear, even if defaults are acceptable: 

[markup]
  [markup.goldmark.renderer]
    unsafe = false  # set true only if content authors need raw HTML

  [markup.tableOfContents]
    startLevel = 2
    endLevel   = 4

Good TOML flat-key ordering is correct

All flat [params] keys (custom_css, background_color_class) appear before nested subtable headers like [params.announcement] and [params.ananke]. This is the correct TOML ordering — placing a subtable header first silently swallows subsequent flat keys into the wrong table. The CLAUDE.md correctly documents this pitfall, and the config correctly avoids it.

2. Theme & Layout Architecture

Warning Heavy Ananke dependency with near-total override

The project declares theme = 'ananke' but overrides virtually every layout. The root layouts/ directory contains 28 HTML files spanning all sections; the only things Ananke provides in practice are: (1) the base template structure (baseof.html), (2) the CSS asset pipeline (GetMainCSS.html), and (3) the menu system. This is a maintenance risk: Ananke updates could break the CSS pipeline or template block contracts without warning.

Recommendation: Consider one of two paths:
• Minimal theme: Replace Ananke with a minimal themes/fenb-base/ that contains only baseof.html, the CSS pipeline partial, and no visual opinions. Eliminates surprise breakage on Ananke updates.
• Keep Ananke, pin the version: Pin the Ananke submodule to a specific commit so updates are controlled. Add a git submodule update step to the release checklist.

The current approach works, but the dependency surface is misleadingly large.

Good Partial system is well-designed and reusable

The custom partials demonstrate clear thinking about reuse:
• event-card.html — single source of truth for event rendering, used on homepage and calendar fallback
• news-card.html — parameterized heading level (h2/h3) and truncation length; correct application of the caller/callee pattern
• section-header.html — one partial drives homepage section headers across four sections
• icon.html — SVG inline renderer with attribute injection; avoids icon fonts and external HTTP requests

The parameter passing via dict is idiomatic Hugo: {{ partial "news-card.html" (dict "page" . "heading" "h3" "truncate" 130) }}.

Issue Category detection by locale-specific string (news/single.html:19)

Results articles load results-table.js conditionally:

{{ if in (slice "Results" "Résultats") .Params.category }}

This checks both English and French strings in a single template, which is fragile. If the French translation of "Results" changes in fr.yaml (or a typo is introduced in a content file), the JS will silently not load for French articles.

Fix: Use a canonical category ID in front matter rather than the display label. Add a category_id: results field (or rename category to carry the ID), and check .Params.category_id. The display label can then be an i18n lookup on the ID, keeping templates locale-agnostic.

Warning Events calendar legend is hardcoded, not data-driven

The category legend in events/list.html (lines 49–56) lists seven hardcoded categories with their CSS swatch classes. If a new event category is added to events.yaml, the legend must be manually updated in the template — a silent omission that is easy to miss.

Fix: Create a data/event_categories.yaml file listing the canonical category IDs. Both the legend and the event card could iterate over it, ensuring they are always in sync. Alternatively, extract the unique categories from hugo.Data.events.events using Hugo's uniq and slice functions at render time.

Warning News sidebar not sorted by date (news/single.html:32)

The recent news sidebar uses:

{{ range first 6 (where .Site.RegularPages "Section" "news") }}

.Site.RegularPages is sorted by date descending by default, but the where filter does not guarantee sort order is preserved in all Hugo versions.

Fix: Be explicit: 

{{ $sortedNews := sort (where .Site.RegularPages "Section" "news") "Date" "desc" }}
{{ range first 6 $sortedNews }}

Good No-JS fallback on the events calendar

The <noscript> block in events/list.html (lines 35–41) renders a full chronological event grid using the event-card partial. This is the correct progressive enhancement pattern: the JS calendar is an enhancement, not a requirement. The schedule page takes this further — it uses Hugo to server-render every event with data-season and data-category attributes, and JavaScript only handles visibility toggling. This gives print support and a no-JS baseline for free.

Info readFile in a loop vs. Hugo Pipes (index.html:78)

Quick-link SVG icons are inlined via readFile inside a range loop:

{{ readFile (printf "static/images/svg/%s" .icon) | safeHTML }}

The icon.html partial (used elsewhere) does the same but with attribute injection. Both call readFile per iteration. Hugo caches file reads within a build, so this is not a correctness problem. However, if the named file is missing, there is no error — the output is silently empty. The icon partial could benefit from a guard:

{{- $svg := readFile (printf "static/images/svg/%s" .name) -}}
{{- if not $svg }}{{ errorf "icon.html: file not found: %s" .name }}{{ end -}}

3. Data Architecture (YAML & hugo.Data)

Good YAML data files as the single source of truth

Events, clubs, board members, hero slides, programs, and join paths all live in data/ and are accessed via hugo.Data.*. Layouts read from data; content files are used only for markdown articles and section index pages. This is the correct separation: structured, maintainable data is in YAML; prose is in Markdown. The consistent snake_case naming convention for files and keys (board_members.yaml → hugo.Data.board_members.members) correctly avoids the Go template dot-notation limitation with hyphens.

Issue Manually entered display_date duplicates the date field

Every event in events.yaml has both a machine-readable date: "2025-09-06" and a manually formatted display_date: "Sept 6, 2025". These can drift (a date gets corrected but the display string is not updated). More critically, French pages receive an English-formatted date string — "Sept 6, 2025" rather than "6 sept. 2025".

Fix: Remove display_date from the YAML schema entirely. In event-card.html, compute the display date from date using Hugo's date formatting and the existing i18n month name lookup the partial already does: 

{{- $dateObj := time .date -}}
{{- $day   := $dateObj.Format "2" -}}
{{- $month := i18n (printf "month_%s" ($dateObj.Format "Jan")) -}}
{{- $year  := $dateObj.Format "2006" -}}
Then compose the locale-appropriate string (day-month-year for French, month-day-year for English) using a simple conditional on $.Site.Language.Lang. This eliminates data duplication and makes French dates actually French.

Good Bilingual data fields use a clear, consistent suffix convention

Bilingual data fields (e.g., role_en/role_fr, name_en/name_fr) use a consistent _en/_fr suffix pattern. Templates access them idiomatically with dynamic key construction:

{{ index . (printf "role_%s" $.Site.Language.Lang) }}

This is the correct Hugo approach for bilingual YAML data that does not warrant full content files. The pattern is applied consistently across board_members.yaml, affiliations, and policies.

Warning ISO dates compared as strings — relies on a format invariant

The homepage upcoming-events filter uses string comparison on dates:

{{ $upcoming := first 4 (where hugo.Data.events.events "date" "ge" $today) }}

This works correctly because ISO 8601 (YYYY-MM-DD) is lexicographically sortable. However, this is an implicit invariant: if someone enters a date in a different format (e.g., Sept 6, 2025), the comparison silently produces wrong results. The YAML schema has no enforcement.

Fix: Either add a comment in events.yaml schema docs making this invariant explicit, or convert dates to time objects before comparing: 

{{ $upcoming := where hugo.Data.events.events "date" "ge" $today }}
is fine as-is — just document the ISO 8601 requirement clearly in README.md.

4. Internationalization & Bilingual Routing

Good Consistent use of relLangURL for all internal links

Every internal link in every template uses | relLangURL. This is the correct Hugo function for multilingual sites — it prepends the language prefix (e.g., /fr/) for non-default languages and leaves the default language URL unchanged. Hardcoded /fr/ prefixes would break language-switching; missing the filter would serve French users English URLs. The consistent application across all 28 layout files is notable.

Good Language switcher handles both translated and untranslated pages

The language switcher in site-header.html first checks .Translations (the current page's translated sibling, if it exists) before falling back to the other language's home page. This prevents dead links when a page only exists in one language — the switcher gracefully degrades to the home page rather than a 404.

Warning Article date in news/single.html is always formatted in English

The article date is rendered as:

<time datetime="...">{{ .Date.Format "January 2, 2006" }}</time>

Hugo's .Date.Format uses Go's time formatting, which always produces English month names ("January", "February", etc.). French articles therefore display "January 2, 2026" instead of "2 janvier 2026".

Fix: Apply the same i18n month lookup already used in event-card.html. Build the date string from parts using i18n (printf "month_%s" ...), then compose it in the appropriate order for the current language.

Info Month name i18n keys are duplicated (abbreviated vs. full)

The i18n files define two sets of month keys: month_Jan (abbreviated, used in event cards) and cal_month_jan (full, used in the calendar JS data). Both sets exist in both language files. This is not wrong, but the two sets serve similar purposes and could be unified — abbreviated months are derivable from full names in most contexts, and the duplication increases translation maintenance overhead.

5. Asset Pipeline & CSS

Good CSS is processed through Hugo Pipes via Ananke's pipeline

Custom CSS files in assets/ananke/css/ are picked up by Ananke's GetMainCSS.html partial, which:
1. Concatenates all CSS into a single resource via resources.Concat
2. Minifies via css.Sass (which handles plain CSS on Hugo Extended)
3. Fingerprints the output in production via resources.Fingerprint for cache busting

Hugo Extended (v0.161.1) is confirmed on this machine, so the pipeline runs correctly. The end result is one minified, versioned CSS file in production — the right approach.

Warning CSS pipeline location is non-obvious to maintainers

CSS files live in assets/ananke/css/ — a path that looks like it belongs to the Ananke theme's internal assets. There is nothing in the README or comments explaining why custom CSS files must be placed in an Ananke-namespaced directory. A future maintainer could reasonably place a new CSS file in static/css/ instead, where it would be served as a separate uncached file rather than being bundled and minified.

Fix: Add a comment to README.md under the CSS conventions section explaining that assets/ananke/css/ is the correct location for custom CSS (not static/), and that files must also be listed in hugo.toml's custom_css array. This is already partly in STYLE_GUIDE.md but deserves explicit explanation of the build mechanism.

Good CSS custom properties enable dark mode without FOUC

fenb-base.css defines all colour tokens as CSS custom properties at :root and overrides them in [data-theme="dark"]. A small inline script in head-additions.html reads localStorage and sets data-theme before the page renders, preventing flash-of-unstyled-content. The system-preference fallback (prefers-color-scheme) is correctly implemented as a media query. This is the modern, correct approach.

Info No SCSS — plain CSS limits future maintainability

All 11 CSS files are plain CSS. Hugo Extended supports SCSS natively via css.Sass, and Ananke's pipeline even handles SCSS files transparently (it checks .MediaType.SubType and compiles them). The project already uses CSS custom properties for variables, so the loss is mainly nesting and mixins. At the current scale (3,500 lines of CSS), this is manageable — but as the site grows, plain CSS becomes harder to maintain. Migrating individual section files to SCSS when they need refactoring would be low-risk.

6. JavaScript Patterns

Issue Most <script> tags lack defer or async

Scripts are loaded at the bottom of the main block, which avoids render-blocking for body content. However, only results-table.js uses defer. hero-slider.js and events-calendar.js have no attribute, meaning they execute synchronously when encountered — blocking the parser until the script downloads and runs.

Fix: Add defer to all <script src="..."> tags. Scripts already placed at the end of main are safe to defer since the DOM they depend on is above them: 

<script src="{{ "js/hero-slider.js" | absURL }}" defer></script>

Warning i18n strings embedded inline in a <script> block as a long array literal

The calendar's window.FENB_CAL object in events/list.html (lines 67–69) embeds month and weekday names as an inline array of i18n calls spread across one very long line. This is valid but makes the template harder to read and update, and couples i18n maintenance to the template.

The event data itself correctly uses | jsonify | safeJS and is handled in JavaScript with a JSON.parse() guard — the CLAUDE.md documents this correctly.

Refactor option: Extract the month/weekday arrays into a dedicated i18n JSON output. Create a data/ template or a Hugo custom output format that emits translated strings as JSON. This keeps the template lean and the translations in the i18n files where they belong.

Good All JavaScript is vanilla, dependency-free, and scoped

No jQuery, no framework, no CDN dependencies. All four JS files (hero-slider.js, events-calendar.js, events-schedule.js, results-table.js) are vanilla JavaScript using IIFEs to avoid global scope pollution. results-table.js conditionally loads only on Results articles, avoiding unnecessary execution. Total JS payload is minimal.

Good Conditional script loading based on content type

results-table.js is loaded only on news articles whose category matches "Results"/"Résultats". This is a correct and efficient pattern — the script is not loaded globally but injected only when needed. The mechanism is a Hugo template conditional, which is evaluated at build time, so there is no runtime overhead.

7. Content Structure & Archetypes

Issue Default archetype is too minimal for a bilingual site

The only archetype is archetypes/default.md with three fields (date, draft, title) in TOML format, while all content files use YAML front matter. It does not enforce:
• Required fields for news articles (category, summary)
• Language-pair creation (easy to create .en.md without .fr.md)
• Correct filename format (mon-dd-title.lang.md)

Fix: Create section-specific archetypes in archetypes/news/: 

---
title: "{{ replace .File.ContentBaseName "-" " " | title }}"
date: {{ .Date }}
draft: true
category: "Announcement"
summary: ""
---
Also, update the default archetype to YAML (removing the TOML +++ delimiters) to match the rest of the content. The bilingual pairing cannot be enforced by archetypes alone — a /fenb-new-news skill already handles this workflow correctly, so the solution is to document that hugo new should not be used directly; the skill should be the entry point.

Good Bilingual content file naming is correct and consistent

News articles use the {mon}-{dd}-{title}.{lang}.md pattern throughout (e.g., apr-05-nb-athletes-nationals.en.md / apr-05-nb-athletes-nationals.fr.md). The dot before the language code is required for Hugo's translation linking — CLAUDE.md correctly documents that a dash would break it. All 10+ article pairs in content/news/2026/ follow this pattern correctly.

Good Layout-driven pages contain explanatory comments

Section index pages whose content is driven by hugo.Data (not the .Content field) contain HTML comments directing editors to the actual data source: 

<!-- Page content is not read from this file. Edit in data/clubs.yaml. -->
This is the correct pattern for preventing confusion when a content editor opens a file expecting to find the displayed text. The CLAUDE.md also documents this convention.

Warning No SEO front matter on news articles

News articles have minimal front matter: title, date, category, summary. There are no description, keywords, or Open Graph fields. Hugo generates a <meta name="description"> from the page's .Description front matter field; without it, search engines see no description tag for news articles.

Fix: Map summary to the meta description. In the head template (or the Ananke head partial), add: 

{{ with .Params.summary }}
  <meta name="description" content="{{ . }}">
{{ else }}
  <meta name="description" content="{{ .Site.Params.description }}">
{{ end }}
No front matter changes needed — summary already exists and contains the right content.

8. Build & Deployment (Makefile)

Good Pagefind integration is correctly wired into the build

The build-prod target runs npx pagefind --site public after Hugo, creating the search index as a post-build step. The dev server pre-builds pagefind before starting (make serve), then uses --renderStaticToDisk so the pagefind files are accessible on disk during development. This is the correct Pagefind integration pattern for Hugo.

Good check-parity target enforces bilingual completeness

The Makefile includes a check-parity target that scans content/ for .en.md files missing their .fr.md partner and vice versa. Running this before a release catches translation gaps. The script exits 0 (non-blocking) so it can be added to CI without failing builds — it reports rather than blocks, which is the right default for a content team that may stage translations separately.

Info No RSS output configured — news articles have no feed

Hugo generates RSS by default, but only if outputs is configured to include "RSS". Without explicit configuration, the news section has no feed. A provincial sports organization's news section is a natural candidate for RSS — coaches and clubs subscribe to updates.

Fix: Add to hugo.toml: 

[outputs]
  home    = ["HTML"]
  section = ["HTML", "RSS"]
  page    = ["HTML"]
Hugo will automatically generate /news/index.xml and /fr/news/index.xml. Add a <link rel="alternate"> in the head template to advertise the feed.

Info Hugo path is hard-coded to /snap/bin/hugo

All Makefile targets use the absolute path /snap/bin/hugo. This breaks for any developer who installed Hugo via a different method (apt, brew, binary download). In CI, it would also fail unless the snap path is explicitly available.

Fix: Use hugo (without the path) and rely on PATH, or define it once as a Makefile variable: 

HUGO ?= /snap/bin/hugo
# or simply:
HUGO := hugo
This lets developers override it with make serve HUGO=/usr/local/bin/hugo.

10. What This Project Does Especially Well

Best Practice Data-driven architecture
Events, clubs, board, programs, and hero content all live in typed YAML data files. Layouts read from hugo.Data; Markdown is reserved for prose. Clean separation of content and structure.
Best Practice Progressive enhancement
Every JS-dependent feature has a working <noscript> fallback. The schedule page server-renders all events as HTML with data attributes; JS adds filtering on top. Accessible and print-friendly.
Best Practice CSS custom properties + dark mode
Entire colour system uses CSS variables. Dark mode via [data-theme="dark"] overrides with a FOUC-prevention inline script. System preference respected as fallback.
Best Practice Consistent relLangURL usage
Every internal link in every template uses | relLangURL. Zero hardcoded language prefixes. Language routing works correctly for both languages throughout the site.
Best Practice Reusable, parameterized partials
news-card.html, event-card.html, section-header.html, and icon.html are cleanly parameterized via dict. No copy-paste rendering code. Single source of truth for each component.
Best Practice Hugo Pipes CSS pipeline
Custom CSS is concatenated, minified (via Ananke's GetMainCSS), and fingerprinted in production. One network request for all styles. Cache-busted automatically on any CSS change.
Best Practice Zero external JS dependencies
All JavaScript is vanilla, scoped with IIFEs, and totals well under 20KB. No CDN dependencies, no framework overhead. Conditional loading of results-table.js only on relevant pages.
Best Practice Bilingual parity enforcement
make check-parity reports missing translation pairs. /fenb-new-news skill creates both language files as a single operation. The workflow makes orphaned single-language articles hard to accidentally create.
